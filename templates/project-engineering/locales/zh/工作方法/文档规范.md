# 文档规范

> 版本：1.1
> 日期：2026-07-21
> 状态：定稿

## 结论在前

项目文档是 AI 与团队可重复调用的项目事实，不是聊天记录的备份。除索引外，每份新建或实质性改写的文档必须使用统一五段式；一个文档只讲一个主题。Agent 必须先读总索引获得导航，再按任务需要打开具体文档，不能在任务开始时批量读取整个工作空间；任务结束后，要更新受影响目录的 `index.md` 状态。

## 正文

### 1. 文档统一结构

除 `index.md` 外，每个项目文档按以下顺序组织：

1. **元信息头**：标题、版本、日期、状态（草稿 / 评审中 / 当前有效 / 已替代 / 已归档）。可补充负责人、适用范围和保鲜元信息，但不可替代四个必填字段；状态的生命周期语义见 [产物保鲜规范](./产物保鲜规范.md)。
2. **结论在前**：3–5 句话，概括结论、适用范围、关键限制或下一步，让读者判断是否需要继续阅读。
3. **正文**：用层级标题组织事实、方案、证据或记录；区分事实、推断和待确认项。
4. **关联引用**：引用其他项目文档时使用相对路径 Markdown 链接，例如 `[项目简报](../项目简报.md)`；不要只写文件名或绝对路径。
5. **变更记录**：按时间倒序记录重要变更，至少包含日期、版本和变更摘要。

### 2. 文档粒度

- 一个文档只回答一个可命名的问题；主题变了就新建文档并从原文链接过去。
- 不创建“所有需求”“全部决策”这类大而全的正文。目录负责导航，专题文档负责内容。
- 当文档超过一个读者能快速定位结论的范围时，先拆分主题，再补充父级索引。

### 3. 索引是导航，不是正文

`index.md` 是例外：它只保留元信息、目录条目、每条一句话简介、状态和相对路径链接；不复制正文、长结论、过程记录或变更细节。目录条目建议使用：`[文档标题](./文件.md) — 一句话简介（状态）`。

### 4. 按需获取与索引维护

1. 开始任务时先读取最近的总索引（通常是 `工作空间/index.md`）获取全局导航。
2. 只读取完成当前任务所需的项目简报、待确认问题和相关专题文档；不要一次性读取所有目录。
3. 进入子目录前先读该目录的 `index.md`，再按条目读取具体文档。
4. 新建、定稿、废弃或实质更新文档后，立即更新其父级 `index.md` 的相对链接、简介、状态和最后更新日期。
5. 任务结束时检查受影响的索引是否仍能帮助下一位读者或 Agent 找到最新事实。
