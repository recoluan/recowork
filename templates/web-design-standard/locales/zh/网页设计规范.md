# 网页设计规范

> 版本：1.0.0
>
> 日期：2026-07-22
>
> 状态：定稿

## 结论在前

本规范为 AI 生成或改造 HTML / Web 页面提供一套克制、现代、可信赖的产品型默认设计语言。优先保证内容层级、真实组件状态、桌面与移动端可用性及可访问性，而不是追求静态截图效果。它适合 SaaS、工具型产品、个人或小团队官网及轻量运营后台；用户已有品牌、设计系统或明确视觉要求时，以用户要求为准。本规范不是完整品牌系统，遇到强品牌艺术站、游戏视觉、复杂电商或严格既有品牌约束时，应先澄清再执行。

## 1. 适用范围与使用方式

### 适用范围

- SaaS、工具型产品、个人或小团队产品官网。
- 轻量运营后台、设置页、数据管理页和任务型工具。
- 在缺少既有设计系统时，为 AI 提供稳定的默认实现约束。

### 不适用范围

- 强品牌艺术站、游戏视觉、复杂电商主站。
- 已有严格品牌手册、组件库或视觉验收稿的项目。
- 需要插画、摄影、3D 或复杂动效作为核心表达的页面。

### 使用方式

1. 在设计、生成或改造页面前，先确认页面目标、目标用户、主任务、内容优先级、已有品牌资产和技术约束。
2. 没有既有规范时，使用本文件的默认 token 与组件规则；有冲突时，用户已有品牌规范优先。
3. 先完成信息结构和关键任务路径，再实现视觉细节；每个关键组件必须覆盖必要状态。
4. 交付前完成第 8 节自检，并在输出中说明检查结果、已知限制与未验证项。

## 2. 设计目标、视觉方向和禁止项

### 设计目标

- 内容优先：页面首先帮助用户理解信息、完成任务和做出判断。
- 层级清晰：通过布局、字号、字重、间距和有限强调建立优先级，不依赖装饰堆叠。
- 克制可信：使用干净的中性色、有限强调色、清晰边界和适度留白。
- 状态真实：交互组件具备默认、悬停、聚焦、禁用、加载、成功或失败等实际需要的状态。
- 可访问性优先：键盘可达、可见焦点、足够对比度、语义结构和触摸目标不可缺失。

### 默认视觉方向

- 背景以白色和浅冷灰为主，正文使用高对比深色，中性色承担辅助信息。
- 主强调色只承担主操作、选中态、链接或关键状态；同一视图内不引入多个竞争性高饱和色。
- 以 1px 边框、轻阴影和 6px 至 8px 圆角建立层级；卡片是内容分组工具，不是整页装饰。
- 留白服务于阅读和扫描。桌面端避免空旷 hero，后台页面避免营销式大标题。

### 禁止项

- 大面积或多层渐变、无意义玻璃拟态、漂浮光球与纯装饰背景。
- 过度圆角、胶囊化按钮泛滥、卡片套卡片或将每个页面区块做成悬浮卡片。
- 营销词堆砌、无法支撑的承诺、虚假数据图表或伪造实时指标。
- 只考虑桌面端、移动端文字溢出、横向滚动或关键操作不可触达。
- 只交付静态视觉：忽略空状态、加载、错误、禁用、焦点和反馈状态。

## 3. 基础 Token

### 色彩

| 用途 | Token | 默认值 |
| --- | --- | --- |
| 页面背景 | `--color-page` | `#FFFFFF` |
| 次级背景 | `--color-surface` | `#F8FAFC` |
| 强调背景 | `--color-surface-strong` | `#EFF6FF` |
| 主文字 | `--color-text` | `#0F172A` |
| 次级文字 | `--color-text-muted` | `#475569` |
| 弱化文字 | `--color-text-subtle` | `#64748B` |
| 边框 | `--color-border` | `#E2E8F0` |
| 主色 | `--color-primary` | `#2563EB` |
| 主色悬停 | `--color-primary-hover` | `#1D4ED8` |
| 成功 | `--color-success` | `#15803D` |
| 警告 | `--color-warning` | `#B45309` |
| 错误 | `--color-danger` | `#B91C1C` |

文字与背景应达到 WCAG AA 对比度；不要仅靠颜色传递成功、失败、选中或必填信息。

### 字体与字号

- 字体栈：`Inter, ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`；中文可回退到系统无衬线字体。
- 正文基准为 `16px / 1.5`；密集表格或辅助说明可用 `14px / 1.5`，不可低于 `12px`。
- 建议层级：`12`, `14`, `16`, `20`, `24`, `32`, `40px`。页面内层级数量应有限，按内容密度选择，不随视口宽度缩放字体。
- 标题字重优先 `600` 或 `700`，正文 `400` 或 `450`，避免用过细字重表达关键信息。

### 间距、容器与尺寸

- 间距基准：`4`, `8`, `12`, `16`, `24`, `32`, `48`, `64px`；相邻元素的间距应反映内容关系。
- 内容容器：桌面最大宽度 `1200px`，常规水平内边距 `24px`；移动端水平内边距 `16px`。
- 交互目标：鼠标与触摸可操作区域至少 `40px` 高或宽，移动端优先达到 `44px`。
- 固定格式组件使用稳定尺寸、网格或最小最大约束，避免状态文字改变布局。

### 圆角、边框与阴影

- 按钮、输入框：`6px`；卡片、弹层：`8px`；标签可为 `999px`，但不将普通按钮做成胶囊。
- 常规边框：`1px solid var(--color-border)`；选中和聚焦必须有可见状态。
- 阴影只用于浮层、下拉菜单和需要明确抬升的元素：`0 8px 24px rgba(15, 23, 42, 0.12)`；普通卡片优先使用边框而非阴影。

## 4. PC / Mobile 响应式策略

- 以内容断点而不是设备名称设计。`768px` 以下默认单列；复杂双栏在空间不足时转为单列；超宽屏保持容器最大宽度。
- PC 端适合并列信息、侧边导航和表格扫描；移动端优先主任务、核心状态和连续阅读，次要信息可下移、折叠或进入详情。
- 导航在移动端必须有清晰、可关闭的入口；不要简单缩小桌面菜单。
- 表格应优先展示关键列；无法压缩时提供横向滚动容器、摘要卡片或详情视图，并标明可滚动区域。
- 表单移动端单列、标签在输入前、错误紧邻字段；不要依赖悬停说明关键操作。
- 不使用仅桌面鼠标悬停可发现的关键内容。焦点顺序、触摸区域和安全区要被检查。

## 5. 页面结构与信息层级规则

- 每页先明确一个主任务和一个主操作。主操作通常只有一个视觉主色按钮。
- 页面结构优先为：全局导航或上下文导航、页面标题和关键状态、主内容区、次要信息或辅助操作。
- 标题表达页面对象或当前任务，不使用空泛口号；支持文案说明价值、限制或下一步。
- 使用语义化 HTML：`header`、`nav`、`main`、`section`、`article`、`form`、`button`、`table` 等匹配真实结构。
- 后台与工具页保持紧凑、有序、便于扫描。不要为普通管理操作创造营销 hero 或无意义大图。
- 内容组之间使用间距和标题分层；仅在内容确实需要边界时使用卡片。避免卡片内再嵌卡片。

## 6. 核心组件规范

### 导航

- 当前项有文字、颜色或底边等明确选中标识，不能只靠位置猜测。
- 桌面与移动导航均支持键盘访问和清晰焦点；移动菜单可打开、关闭并保留当前上下文。

### 按钮

- 区分主按钮、次按钮和危险操作；一个区域只保留一个主按钮。
- 使用熟悉图标的工具操作可用图标按钮，并提供 `aria-label` 或可见提示；图标不代替不熟悉的动作文字。
- 覆盖默认、悬停、焦点、禁用和加载状态；加载中避免重复提交，禁用必须仍可辨识。

### 表单

- 每个字段有可见标签、合理默认值或占位说明、必填标识和字段级错误信息。
- 错误应说明如何修复，不只显示红色边框；提交结果应有页面级反馈。
- 不把占位文本当作唯一标签；支持键盘、自动填充和可访问名称。

### 卡片

- 用于重复条目、可独立扫描的内容或有明确边界的工具面板。
- 卡片内保留稳定内边距、有限标题层级和一个清楚的主要操作；避免整页都由装饰卡片构成。

### 列表与表格

- 列表适合阅读顺序，表格适合横向比较。数据表应有列标题、对齐规则、空状态和加载状态。
- 数字、日期和状态保持一致格式；重要状态同时使用文字和颜色或图标。
- 批量操作、排序、筛选和分页只在真实需求存在时加入，避免制造静态假功能。

### 空状态与反馈状态

- 空状态说明当前没有什么、为什么可能为空，以及可以采取的下一步。
- 加载状态保留布局并明确正在加载；错误状态给出可执行的重试或恢复路径。
- 成功、警告和错误提示应可感知、可关闭或自动消失的行为可预测，不能遮挡关键内容。

## 7. AI 实现指令

1. 先读取本规范和项目已有品牌、组件库、页面代码与技术约束；既有品牌或明确用户要求优先。
2. 信息不完整时，先提出最少且关键的澄清问题；不要把猜测包装为需求事实。
3. 先提出简短实现计划：页面主任务、信息层级、响应式处理、组件状态与验证方式。范围重大或视觉方向不明确时，等待确认后再大改。
4. 实现真实 HTML、CSS 与交互状态。不要只输出截图式结构、虚假图表或不可用按钮。
5. 保持改动聚焦，复用项目既有组件和 token；确有必要才新增抽象或依赖。
6. 完成后按第 8 节逐项检查，并报告通过项、修复项、已知限制和未验证事项。

## 8. 页面实现后的自检清单

- [ ] 已确认页面目标、用户、主任务、内容优先级和已有品牌要求。
- [ ] 用户已有品牌、设计系统或明确视觉要求已优先执行；偏离默认规范的原因已说明。
- [ ] 信息层级清楚，主操作唯一明确，正文和标题没有溢出或互相遮挡。
- [ ] 色彩、字号、间距、容器、圆角、边框和阴影符合本规范或已有设计系统。
- [ ] 已检查桌面与移动端布局、导航、表单、表格或列表的适配，不依赖仅悬停可见的信息。
- [ ] 按钮、表单和关键组件具备需要的默认、悬停、焦点、禁用、加载、空、成功或错误状态。
- [ ] 键盘操作、焦点可见性、语义结构、图片替代文本和文字对比度已检查。
- [ ] 未使用无意义渐变、玻璃拟态、过度圆角、虚假数据或仅静态的装饰效果。
- [ ] 已运行项目可用的构建、测试或视觉检查，并在交付说明中报告结果。
