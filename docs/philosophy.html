设计理念

让 AI 用工程化的方式聪明地工作。

RecoWork 不把 AI 当成一次性聊天窗口，而是把角色、流程、知识、质量和使用位置组合成可复用系统。

不是 Prompt 集合

Prompt 只是入口。真正可复用的是角色协议、工作流程、文档规范、检查清单和续聊记忆。

模板和 target 分离

模板描述工作方法，target 描述输出位置。这样同一个项目工程化模板可以生成 Codex、Claude Code 或 Notion 所需资产。

尊重真实工具约定

Claude Code 使用 CLAUDE.md 和 .claude/skills；Codex 使用 AGENTS.md。RecoWork 不发明看起来整齐但工具并不认识的目录。

面向持续协作

AI 不应该每次从零猜上下文。索引、工作空间和记忆卡让下一次协作从已有事实继续。

核心模型

template + target + locale = initialized workflow

Template

回答：要工程化什么工作场景？例如项目协作、系统学习、日常 AI 使用。

Target

回答：这些资产要放到哪里使用？例如聊天窗口、本地代码项目、Notion 工作区或飞书文档。

Locale

回答：用户看到的语言和文件夹命名是什么？语言可以本地化，但约定文件名保持稳定。

反模式

RecoWork 有意避免这些 AI 使用方式。

每次重新解释背景

聊天窗口里的临时说明很难持续维护。RecoWork 用工作空间、index.md 和续聊记忆把上下文沉淀下来。

把品牌名当 target

Claude Chat 和 Claude Code 的使用方式完全不同。RecoWork 使用具体 target，避免把聊天说明和项目配置混在一起。

只追求更长 Prompt

长 Prompt 不等于稳定工作。稳定来自明确角色、边界、检查标准、文档结构和可恢复的记忆。

发明工具不认识的目录

项目 target 必须遵守真实工具约定。看起来整齐但工具不会读取的目录，会制造伪工程化。

设计取舍

RecoWork 优先选择可持续，而不是一次性惊艳。

01 少一点魔法，多一点可检查

初始化结果是文件和 Prompt，用户可以打开、审阅、修改和提交。

02 先支持真实使用面

Target ID 描述具体使用面：chat、project、workspace 或 doc，而不是泛泛的平台品牌。

03 让模板内容持续演进

模板只描述工作方法；target 负责输出约定。二者分离后，新增工具不需要复制所有模板。

RecoWork 面向真实 AI 工作的开源工作流模板。
